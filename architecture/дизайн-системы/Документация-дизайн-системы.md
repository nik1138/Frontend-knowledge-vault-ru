# Документация-дизайн-системы

## Обзор

Документация дизайн-системы — это критически важный элемент, обеспечивающий успешное внедрение, использование и поддержку системы. Хорошо структурированная документация помогает командам понимать, как использовать компоненты, следовать принципам дизайна и эффективно внедрять элементы системы в свои проекты.

Этот документ охватывает подходы к созданию документации дизайн-системы, структуру документации, инструменты и лучшие практики для фронтенд-разработчиков и дизайнеров.

## Важность документации дизайн-системы

### Ключевые функции документации

1. **Обучение пользователей** - помогает новым членам команды быстро освоить систему
2. **Стандартизация** - обеспечивает единообразное использование компонентов
3. **Поддержка принятия решений** - предоставляет рекомендации по выбору компонентов
4. **Снижение порога входа** - упрощает интеграцию дизайн-системы в проекты
5. **Сохранение знаний** - фиксирует архитектурные и дизайнерские решения

### Целевая аудитория

Документация должна учитывать потребности различных групп пользователей:

- **Фронтенд-разработчики** - нуждаются в технической информации, API и примерах кода
- **Дизайнеры** - требуют визуальных руководств, принципов и ограничений
- **Продуктовые менеджеры** - интересуются возможностями системы и процессами
- **QA-инженеры** - нуждаются в критериях тестирования и состояниях компонентов

## Структура документации дизайн-системы

### 1. Главная страница и обзор

Главная страница должна содержать:

- Краткое описание дизайн-системы
- Цели и преимущества системы
- Контактную информацию команды
- Быстрые ссылки на ключевые разделы
- Статус системы (стабильная, бета, экспериментальная)

```markdown
# Дизайн-система "Альфа"

Добро пожаловать в дизайн-систему "Альфа" - унифицированный набор компонентов, 
принципов и руководств для создания согласованных пользовательских интерфейсов.

## Цели системы
- Обеспечение консистентности интерфейсов
- Ускорение процесса разработки
- Повышение доступности продуктов
- Упрощение масштабирования

## Начало работы
- [Установка и настройка](getting-started/installation)
- [Первые шаги](getting-started/quickstart)
- [Принципы дизайна](principles)

## Статус
Текущая версия: v2.4.0 (стабильная)
Последнее обновление: 15 ноября 2023
```

### 2. Принципы и философия

Раздел, описывающий основные принципы системы:

```markdown
# Принципы дизайна

## Единство
Все компоненты следуют единым принципам и визуальному языку.

## Доступность
Каждый компонент соответствует стандартам WCAG 2.1 AA.

## Гибкость
Компоненты адаптируются к различным контекстам использования.

## Масштабируемость
Система легко расширяется новыми компонентами и возможностями.
```

### 3. Компоненты

Для каждого компонента должна быть отдельная страница с полной документацией:

#### Структура документации компонента

```markdown
# Button (Кнопка)

[[Компоненты-дизайна]]

Кнопка - интерактивный элемент управления, который позволяет пользователю выполнить действие или выбрать опцию.

## Варианты использования

- Первичные действия (primary)
- Второстепенные действия (secondary)
- Опасные действия (danger)
- Текстовые кнопки (text)

## Варианты

### Primary
Используется для основных действий на странице.

<div class="component-preview">
  <button class="btn btn--primary">Primary Button</button>
</div>

```jsx
<Button variant="primary" onClick={handleClick}>
  Primary Action
</Button>
```

### Secondary
Используется для второстепенных действий.

<div class="component-preview">
  <button class="btn btn--secondary">Secondary Button</button>
</div>

```jsx
<Button variant="secondary" onClick={handleClick}>
  Secondary Action
</Button>
```

## API

| Свойство | Тип | Значение по умолчанию | Описание |
|----------|-----|----------------------|----------|
| variant | 'primary' \| 'secondary' \| 'tertiary' \| 'danger' | 'primary' | Вариант стиля кнопки |
| size | 'small' \| 'medium' \| 'large' | 'medium' | Размер кнопки |
| loading | boolean | false | Состояние загрузки |
| disabled | boolean | false | Отключить кнопку |
| icon | ReactNode | - | Иконка слева от текста |
| fullWidth | boolean | false | Занимать всю доступную ширину |

## Состояния

- **Нормальное** - обычное состояние компонента
- **Наведение** - состояние при наведении курсора
- **Фокус** - состояние при получении фокуса
- **Активное** - состояние при нажатии
- **Загрузка** - состояние ожидания
- **Отключено** - неактивное состояние

## Доступность

Компонент соответствует требованиям WCAG 2.1:
- Имеет корректную семантику (тег button)
- Поддерживает клавиатурную навигацию
- Имеет достаточный контраст
- Правильно обрабатывает ARIA-атрибуты

## Рекомендации по использованию

- Используйте primary кнопки для главных действий
- Не используйте более одной primary кнопки в одной области
- Добавляйте иконки только если они улучшают понимание
- Избегайте слишком длинных текстов на кнопках
```

    },
  },

export const Primary = Template.bind({});
Primary.args = {
  variant: 'primary',
  children: 'Primary Button',
};

export const Secondary = Template.bind({});

  variant: 'secondary',
  children: 'Secondary Button',

  children: 'Send Message',

  variant: 'primary',
  loading: true,
  children: 'Loading...',

### 5. Сравнение и рекомендации

Для похожих компонентов необходимы сравнения:

```markdown
# Button vs Link

| Характеристика | Button | Link |
|----------------|--------|------|
| Назначение | Выполнение действия | Навигация |
| Семантика | `<button>` | `<a>` |
| Стили | Акцентные | Менее акцентные |
| Использование | При отправке форм | При переходе по ссылкам |

## Когда использовать

**Button** - когда действие не ведет к новой странице
**Link** - когда происходит переход между страницами
```

## Поддержка документации

### 1. Версионирование

Документация должна соответствовать версиям дизайн-системы:

```markdown
> **Версия:** Доступно с v2.1.0
> **Депрекейтед:** Устарело в v2.5.0, будет удалено в v3.0.0
```

### 2. Автоматическое обновление

Используйте CI/CD для автоматического обновления документации:

```yaml
# .github/workflows/deploy-docs.yml
name: Deploy Docs
on:
  push:
    branches: [main]
    paths: ['docs/**', 'src/**']

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Setup Node.js
        uses: actions/setup-node@v2
        with:
          node-version: '16'
      - name: Install dependencies
        run: npm ci
      - name: Build documentation
        run: npm run build:docs
      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./build
```

### 3. Обратная связь

Предоставьте пользователям возможность оставлять отзывы:

```jsx
// FeedbackWidget.jsx
import React, { useState } from 'react';

const FeedbackWidget = () => {
  const [feedback, setFeedback] = useState('');
  const [submitted, setSubmitted] = useState(false);

  const handleSubmit = (e) => {
    e.preventDefault();
    // Отправка отзыва
    console.log('Feedback:', feedback);
    setSubmitted(true);
  };

  if (submitted) {
    return <div>Спасибо за ваш отзыв!</div>;
  }

  return (
    <form onSubmit={handleSubmit}>
      <textarea
        value={feedback}
        onChange={(e) => setFeedback(e.target.value)}
        placeholder="Как мы можем улучшить эту документацию?"
      />
      <button type="submit">Отправить</button>
    </form>
  );
};
```

## Заключение

Документация дизайн-системы играет ключевую роль в успешном внедрении и использовании системы. Хорошо структурированная, содержательная и актуальная документация помогает командам эффективно работать с дизайн-системой, снижает порог входа для новых участников и обеспечивает консистентность в использовании компонентов.

Для создания эффективной документации необходимо использовать подходящие инструменты, следовать лучшим практикам и регулярно обновлять информацию в соответствии с изменениями в дизайн-системе.

Для более глубокого понимания компонентного подхода рекомендуется ознакомиться с [[Компоненты-дизайна]] и [[Поддержка-дизайн-системы]].
