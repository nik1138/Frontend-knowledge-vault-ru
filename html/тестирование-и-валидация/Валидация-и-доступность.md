# Валидация-и-доступность

## Обзор

Валидация HTML для доступности - это процесс проверки HTML кода с точки зрения его соответствия требованиям доступности для пользователей с ограниченными возможностями. В условиях российской цифровой среды 2025 года, где требования к доступности становятся обязательными по федеральному законодательству, корректная валидация HTML с учетом принципов доступности приобретает особое значение.

## Основы доступности в HTML

### Принципы доступности

Доступность веб-контента основывается на следующих принципах (POUR):

- **Perceivable (Воспринимаемость)** - информация должна быть представлена способами, доступными для восприятия
- **Operable (Управляемость)** - интерфейсные компоненты должны быть управляемыми
- **Understandable (Понятность)** - информация и работа пользовательского интерфейса должны быть понятными
- **Robust (Надежность)** - контент должен быть надежным, то есть достаточно прочным, чтобы различные вспомогательные технологии могли его интерпретировать

### Законодательная база в России

В 2025 году в России действуют следующие нормативные акты:

- Федеральный закон №419-ФЗ от 27.12.2018 (о внесении изменений в 254-ФЗ)
- ГОСТ Р 52872-2019 "Интернет-ресурсы. Общие эргономические требования"
- Требования к созданию доступной среды для лиц с ограниченными возможностями

## Связь валидации и доступности

### Влияние валидности на доступность

Корректная HTML валидация напрямую влияет на доступность:

- Валидный HTML легче интерпретируется вспомогательными технологиями
- Семантически правильная разметка улучшает навигацию для пользователей скринридеров
- Отсутствие синтаксических ошибок предотвращает искажение структуры документа
- Правильное вложение элементов обеспечивает корректное понимание иерархии контента

### Стандарты WCAG 2.1

Наиболее актуальная версия руководящих принципов доступности веб-контента:

- **Уровень A** - минимальный уровень доступности
- **Уровень AA** - рекомендуемый уровень для большинства сайтов
- **Уровень AAA** - наивысший уровень доступности

## Ключевые элементы доступности

### Альтернативный текст

Каждое изображение должно иметь альтернативный текст:

```html
<!-- Хорошо -->
<img src="chart.jpg" alt="График роста продаж за последние 5 лет">

<!-- Для декоративных изображений -->
<img src="decor.jpg" alt="" role="presentation">

<!-- Для функциональных изображений -->
<button>
    <img src="delete-icon.png" alt="Удалить элемент">
</button>
```

### Семантическая структура

Правильная структура документа важна для пользователей вспомогательных технологий:

```html
<!DOCTYPE html>
<html lang="ru">
<head>
    <meta charset="UTF-8">
    <title>Доступная страница</title>
</head>
<body>
    <header>
        <h1>Название сайта</h1>
        <nav aria-label="Основная навигация">
            <ul>
                <li><a href="/">Главная</a></li>
                <li><a href="/about">О нас</a></li>
            </ul>
        </nav>
    </header>
    
    <main>
        <h2>Основной заголовок страницы</h2>
        <p>Контент страницы...</p>
    </main>
    
    <aside aria-label="Дополнительная информация">
        <h3>Сопутствующая информация</h3>
        <!-- Содержимое -->
    </aside>
    
    <footer>
        <p>Информация о сайте</p>
    </footer>
</body>
</html>
```

### Формы и интерактивные элементы

## Инструменты проверки доступности

### axe-core

Один из самых популярных инструментов для проверки доступности:

```javascript
const axe = require('axe-core');

// Проверка доступности
axe.run(document, {
    runOnly: {
        type: 'tag',
        values: ['wcag2a', 'wcag2aa', 'section508']
    }
}, (err, results) => {
    if (err) throw err;
    console.log('Нарушения доступности:', results.violations);
});
```

### Lighthouse

Интегрированная проверка доступности в DevTools и CLI:

```bash
# Проверка доступности через CLI
lighthouse https://example.com --only-categories=accessibility
```

### WAVE

Инструмент для веб-анализа доступности:

- Расширение для браузеров
- Онлайн-сервис
- Детальный отчет о проблемах доступности

### Pa11y

Автоматизированный инструмент проверки доступности:

```bash
# Установка
npm install -g pa11y

# Проверка URL
pa11y https://example.com

# Проверка с определенными стандартами
pa11y https://example.com --standard WCAG2AA
```

## Практические правила доступности

### Цвет и контрастность

- Минимальный коэффициент контрастности 4.5:1 для обычного текста
- 3:1 для крупного текста
- Не использовать цвет как единственный способ передачи информации

```css
/* Пример обеспечения контраста */
.high-contrast-text {
    color: #000;
    background-color: #fff;
    /* Коэффициент контрастности: 21:1 */
}
```

### Клавиатурная навигация

- Все интерактивные элементы должны быть доступны через клавиатуру
- Визуальная индикация фокуса
- Логический порядок навигации

```css
/* Обеспечение видимого фокуса */
button:focus,
input:focus,
a:focus {
    outline: 2px solid #005fcc;
    outline-offset: 2px;
}
```

### Явные метки

- Все элементы форм должны иметь связанные метки
- Использование aria-атрибутов при необходимости

```html
<!-- Явная метка -->
<label for="username">Имя пользователя</label>
<input type="text" id="username" name="username">

<!-- Альтернатива через aria-label -->
<input type="text" aria-label="Имя пользователя" name="username">

<!-- Альтернатива через aria-labelledby -->
<h3 id="username-label">Имя пользователя</h3>
<input type="text" aria-labelledby="username-label" name="username">
```

## Автоматизация проверки доступности

### Интеграция в CI/CD

```yaml
# GitHub Actions для проверки доступности
name: Accessibility Check
on: [push, pull_request]

jobs:
  accessibility:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '18'
    - name: Install pa11y
      run: npm install -g pa11y
    - name: Run accessibility tests
      run: pa11y --standard WCAG2AA --ignore notice,url --threshold 0 http://localhost:3000
```

### Проверка в тестах

```javascript
// Jest тесты для проверки доступности
const { AxePuppeteer } = require('@axe-core/puppeteer');
const puppeteer = require('puppeteer');

describe('Доступность веб-страницы', () => {
    let browser;
    let page;

    beforeAll(async () => {
        browser = await puppeteer.launch();
        page = await browser.newPage();
        await page.goto('http://localhost:3000');
    });

    afterAll(async () => {
        await browser.close();
    });

    test('Страница не должна содержать серьезных нарушений доступности', async () => {
        const results = await new AxePuppeteer(page)
            .withTags(['wcag2a', 'wcag2aa'])
            .analyze();

        expect(results.violations).toHaveLength(0);
    });
});
```

## Проверка различных типов контента

### Мультимедийный контент

- Предоставление субтитров для видео
- Аудиоописание для видео
- Текстовые альтернативы для аудио

```html
<video controls>
    <source src="video.mp4" type="video/mp4">
    <track kind="subtitles" src="subtitles-ru.vtt" srclang="ru" label="Русский">
    <track kind="subtitles" src="subtitles-en.vtt" srclang="en" label="English">
    Ваш браузер не поддерживает видео тег.
</video>
```

### Интерактивные элементы

- Корректное использование ролей (roles)
- Правильные состояния (states)
- Соответствующие атрибуты (properties)

```html
<button 
    aria-expanded="false" 
    aria-controls="collapsible-content"
    onclick="toggleContent()">
    Показать детали
</button>

<div id="collapsible-content" hidden>
    Скрытый контент
</div>
```
