# Тестирование доступности HTML

Тестирование доступности HTML — это процесс проверки соответствия веб-страниц стандартам доступности, чтобы убедиться, что контент доступен для всех пользователей, включая людей с ограниченными возможностями. Доступность (accessibility) позволяет пользователям с различными нарушениями воспринимать, понимать, навигировать и взаимодействовать с веб-сайтами.

## Основные аспекты доступности

Доступность веб-контента регулируется стандартами WCAG (Web Content Accessibility Guidelines), которые определяют четыре основных принципа:

- **Воспринимаемость** (Perceivable) — информация должна быть представлена в формате, доступном для восприятия всеми пользователями
- **Доступность** (Operable) — интерфейс должен быть удобен в управлении для всех пользователей
- **Понятность** (Understandable) — информация и работа интерфейса должны быть понятны
- **Надежность** (Robust) — контент должен быть надежным и работать с различными вспомогательными технологиями

Для понимания основ семантической структуры HTML рекомендуется ознакомиться с [[html/semantic-structure]].

## Инструменты для тестирования доступности

### Автоматические инструменты

Автоматические инструменты позволяют быстро выявить большую часть проблем доступности. Они интегрируются в браузеры, IDE и сборочные процессы.

- **axe-core** — популярная библиотека для автоматического тестирования доступности, доступна как расширение для Chrome и Firefox
- **Lighthouse** — встроенный инструмент в Chrome DevTools, позволяет анализировать доступность в контексте других метрик
- **WAVE** — веб-сервис и расширение для браузеров, показывает визуальные индикаторы проблем
- **Pa11y** — инструмент командной строки для автоматического тестирования, может быть интегрирован в CI/CD

### Ручные тесты

Ручное тестирование не может быть заменено автоматическими инструментами полностью, так как требует человеческого восприятия и понимания контекста.

- Проверка клавиатурной навигации без использования мыши
- Тестирование с использованием скринридера
- Проверка цветового контраста визуальных элементов
- Анализ логики семантической разметки

## Проверка семантики

Семантическая разметка — это основа доступности. Правильное использование HTML-тегов помогает вспомогательным технологиям понимать структуру документа.

### Пример правильной семантики:

```html
<header>
  <h1>Заголовок страницы</h1>
  <nav aria-label="Основная навигация">
    <ul>
      <li><a href="/">Главная</a></li>
      <li><a href="/about">О нас</a></li>
    </ul>
  </nav>
</header>

<main>
  <article>
    <h2>Статья</h2>
    <p>Содержимое статьи...</p>
  </article>
</main>

<aside>
  <h3>Боковая панель</h3>
  <p>Дополнительная информация</p>
</aside>

<footer>
  <p>© 2023 Все права защищены</p>
</footer>
```

Важно использовать заголовки в правильной иерархии и избегать пропусков уровней (например, сразу переходить от h1 к h3). Для более подробного изучения семантической структуры см. [[html/semantic-structure]].

## Проверка клавиатурной навигации

Клавиатурная навигация важна для пользователей, которые не могут использовать мышь. Все интерактивные элементы должны быть доступны с помощью клавиатуры.

### Требования:

- Все интерактивные элементы (ссылки, кнопки, формы) должны быть доступны через Tab
- Порядок фокуса должен быть логичным и последовательным
- Визуальный индикатор фокуса должен быть виден (outline)
- Фокус не должен застревать на каком-либо элементе

### Пример стилизации фокуса:

```css
button:focus,
input:focus,
a:focus {
  outline: 2px solid #005fcc;
  outline-offset: 2px;
}
```

Для изучения взаимодействия с формами и доступности элементов управления см. [[html/forms-accessibility]].

## Проверка скринридеров

Скринридеры — это программное обеспечение, которое преобразует текст на экране в речь или шрифт Брайля. Проверка с их использованием помогает понять, как вспомогательные технологии воспринимают ваш контент.

### Популярные скринридеры:

- **NVDA** — бесплатный скринридер для Windows
- **JAWS** — коммерческий скринридер для Windows
- **VoiceOver** — встроенный скринридер в macOS и iOS
- **TalkBack** — для Android устройств

При тестировании с помощью скринридера обращайте внимание на:

- Правильное объявление элементов
- Четкость и полноту альтернативного текста для изображений
- Логичную структуру навигации по заголовкам и спискам
- Правильное объявление состояний (например, открыт/закрыт для аккордеонов)

## Лучшие практики тестирования

1. **Комбинируйте автоматические и ручные тесты** — автоматика покрывает до 30-40% проблем, остальные требуют ручной проверки
2. **Тестируйте регулярно** — включайте проверки в процесс разработки, а не только в конце
3. **Используйте реальные сценарии** — проверяйте, как пользователи на самом деле будут использовать ваш интерфейс
4. **Работайте с людьми с ограниченными возможностями** — их опыт бесценен для понимания реальных потребностей
5. **Документируйте результаты** — ведите учет проблем и их решений для будущих проектов

## Примеры кода для улучшения доступности

### Альтернативный текст для изображений:

```html
<img src="chart.png" alt="График роста продаж за последние 6 месяцев">
```

### Явные метки для форм:

```html
<label for="email">Email:</label>
<input type="email" id="email" name="email" required>
```

### ARIA-атрибуты для сложных компонентов:

```html
<button aria-expanded="false" aria-controls="menu">
  Меню
</button>
<div id="menu" role="menu" aria-hidden="true">
  <!-- Пункты меню -->
</div>
```

Для более глубокого изучения ARIA и ролей см. [[html/aria-roles]].

## Заключение

Тестирование доступности — это не одноразовая задача, а постоянный процесс, который должен сопровождать весь цикл разработки. Совмещение автоматических и ручных проверок, регулярное тестирование с использованием скринридеров и других вспомогательных технологий позволяет создавать более инклюзивные веб-сайты.
