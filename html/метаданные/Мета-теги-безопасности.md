# Мета-теги-безопасности

Мета-теги безопасности - это специальные HTML-теги, которые помогают защитить веб-сайты от различных атак, таких как межсайтовый скриптинг (XSS), кликджекинг и другие уязвимости. Эти теги устанавливают политики безопасности, которые браузеры применяют при загрузке и выполнении веб-страницы.

## Основные мета-теги безопасности

### Content Security Policy (CSP)
```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' 'unsafe-inline' https://analytics.example.com; style-src 'self' 'unsafe-inline';">
```
CSP помогает предотвратить атаки типа "межсайтовый скриптинг" (XSS) путем ограничения источников, с которых могут загружаться ресурсы.

#### Основные директивы CSP:
- `default-src` - политика по умолчанию для всех ресурсов
- `script-src` - источники для JavaScript
- `style-src` - источники для CSS
- `img-src` - источники для изображений
- `connect-src` - источники для AJAX и fetch запросов
- `font-src` - источники для шрифтов
- `object-src` - источники для плагинов
- `media-src` - источники для аудио и видео

> [!tip] Рекомендация
> Для российских сайтов в 2025 году особенно важно использовать CSP для защиты от XSS-атак, которые могут использоваться для кражи данных пользователей.

### X-UA-Compatible
```html
<meta http-equiv="X-UA-Compatible" content="IE=edge">
```
Указывает, каким образом Internet Explorer должен отображать страницу. Важно для обеспечения совместимости и безопасности в устаревших браузерах.

### Referrer Policy
```html
<meta name="referrer" content="no-referrer">
```
Контролирует, какая информация об отправителе передается при переходе по ссылкам.

#### Значения Referrer Policy:
- `no-referrer` - не отправлять реферер
- `no-referrer-when-downgrade` - не отправлять при переходе с HTTPS на HTTP
- `same-origin` - отправлять только для запросов в пределах одного источника
- `strict-origin` - отправлять только домен при переходе с HTTPS на HTTP
- `strict-origin-when-cross-origin` - отправлять полный URL для одного источника, только домен при переходе между источниками и только при HTTPS

## Российские особенности 2025

В 2025 году в России:

- Увеличение требований к защите персональных данных (ФЗ-152)
- Рост числа кибератак на веб-ресурсы
- Ужесточение требований к безопасности сайтов
- Развитие отечественных браузеров и систем безопасности

### Соответствие требованиям ФЗ-152

Для сайтов, обрабатывающих персональные данные:
```html
<meta http-equiv="Content-Security-Policy" content="upgrade-insecure-requests">
<meta name="referrer" content="strict-origin-when-cross-origin">
```

### Защита от утечки данных

Использование CSP для предотвращения внедрения вредоносных скриптов:
```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' https://trusted-cdn.com; object-src 'none'; form-action 'self'; frame-ancestors 'none';">
```

## Расширенные мета-теги безопасности

### X-Frame-Options (через HTTP заголовки, но может быть указан через meta)
```html
<meta http-equiv="X-Frame-Options" content="DENY">
```
Предотвращает встраивание страницы в iframe на других сайтах, защищая от кликджекинга.

### X-Content-Type-Options
```html
<meta http-equiv="X-Content-Type-Options" content="nosniff">
```
Предотвращает MIME-type sniffing, когда браузер пытается определить тип файла по его содержимому, а не по заголовкам.

### Strict-Transport-Security
```html
<meta http-equiv="Strict-Transport-Security" content="max-age=31536000; includeSubDomains; preload">
```
Принудительно использует HTTPS для всех соединений с сайтом.

> [!warning] Важно
> HSTS через meta-тег не поддерживается всеми браузерами. Рекомендуется использовать HTTP-заголовки для этой директивы.

## Практические рекомендации

### Политика безопасности для типичного сайта
```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' 'unsafe-inline' https://analytics.google.com https://api.example.com; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self'; connect-src 'self' https://api.example.com; frame-ancestors 'none';">
<meta name="referrer" content="strict-origin-when-cross-origin">
<meta http-equiv="X-Content-Type-Options" content="nosniff">
<meta http-equiv="X-Frame-Options" content="DENY">
```

### Постепенное внедрение CSP
1. Начните с `Content-Security-Policy-Report-Only` для тестирования:
```html
<meta http-equiv="Content-Security-Policy-Report-Only" content="default-src 'self'; report-uri /csp-report">
```
2. Соберите отчеты об ошибках
3. Настройте политику под ваш сайт
4. Переключитесь на полноценный CSP

### Тестирование безопасности

Используйте инструменты для проверки мета-тегов безопасности:
- CSP Evaluator от Google
- Security Headers (securityheaders.com)
- Mozilla Observatory
- Яндекс.Вебмастер (для российских сайтов)

## Совместимость с браузерами

- **Chrome**: Полная поддержка всех основных мета-тегов безопасности
- **Firefox**: Полная поддержка
- **Safari**: Полная поддержка
- **Edge**: Полная поддержка
- **Yandex Browser**: Полная поддержка (важно для российских пользователей)

## Ошибки при реализации

1. **Слишком строгая CSP**: Блокировка необходимых скриптов и ресурсов
2. **Слишком слабая CSP**: Недостаточная защита
3. **Неправильные пути ресурсов**: Указание на неправильные домены
4. **Отсутствие тестирования**: Внедрение без проверки на реальном сайте
5. **Игнорирование отчетов**: Не настройка отчетов о нарушениях CSP

> [!tip] Совет
> В России в 2025 году особенно важно следить за безопасностью веб-сайтов из-за увеличения числа киберугроз и ужесточения законодательства в области защиты данных.

## Современные тенденции безопасности

### Subresource Integrity (SRI)
Хотя это не мета-тег, но важная мера безопасности:
```html
<script src="https://cdn.example.com/script.js" integrity="sha384-..." crossorigin="anonymous"></script>
```

### Feature Policy (теперь Permissions Policy)
```html
<meta http-equiv="Permissions-Policy" content="geolocation=(), microphone=(), camera=()">
```
