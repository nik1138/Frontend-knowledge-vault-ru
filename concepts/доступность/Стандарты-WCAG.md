# Стандарты WCAG

## Обзор

Руководящие принципы доступности веб-контента (Web Content Accessibility Guidelines, WCAG) представляют собой международные стандарты, разработанные для обеспечения доступности веб-контента для людей с ограниченными возможностями. Эти руководящие принципы помогают разработчикам создавать веб-сайты, которые могут использоваться всеми, включая людей с нарушениями зрения, слуха, двигательных функций и когнитивных способностей.

## История и версии

WCAG были разработаны Консорциумом Всемирной паутины (W3C) в рамках Инициативы доступности веб-содержимого (WAI). Существует несколько версий:

- **WCAG 1.0** - первая версия, опубликованная в 1999 году
- **WCAG 2.0** - опубликованная в 2008 году, до сих пор широко используется
- **WCAG 2.1** - дополнение 2018 года, добавляющее мобильные и когнитивные аспекты
- **WCAG 2.2** - дополнение 2023 года, включающее дополнительные рекомендации по когнитивной доступности
- **WCAG 3.0** - в разработке, представляет собой радикальное обновление подхода к оценке доступности

## Четыре основных принципа (POUR)

Все рекомендации WCAG основаны на четырех основных принципах:

### 1. Воспринимаемость (Perceivable)
Информация и компоненты пользовательского интерфейса должны быть представлены способами, которые пользователи могут воспринимать.

**Примеры:**
- Предоставление альтернативного текста для изображений ([[Семантическая-верстка]])
- Предоставление субтитров для аудио и видео контента
- Использование цвета не как единственного способа передачи информации

### 2. Управляемость (Operable)
Компоненты пользовательского интерфейса и навигация должны быть управляемыми.

**Примеры:**
- Обеспечение возможности управления всеми функциями через клавиатуру
- Предоставление достаточного времени для чтения и использования контента
- Предотвращение проблем с эпилепсией и судорогами

### 3. Понятность (Understandable)
Информация и работа пользовательского интерфейса должны быть понятными.

**Примеры:**
- Создание текста, который можно понять
- Обеспечение предсказуемости веб-страниц
- Помощь пользователям избежать и исправить ошибки

### 4. Прочность (Robust)
Контент должен быть достаточно прочным, чтобы быть интерпретируемым различными вспомогательными технологиями.

**Примеры:**
- Использование валидного HTML-кода
- Правильное использование ARIA-атрибутов ([[ARIA-атрибуты]])
- Обеспечение совместимости с текущими и будущими вспомогательными технологиями

## Уровни соответствия

WCAG 2.x определяют три уровня соответствия:

### Уровень A (Минимальный)
Базовый уровень доступности. Игнорирование этих критериев создает барьеры, которые делают определенный контент недоступным для многих пользователей с ограниченными возможностями.

**Пример:**
```html
<!-- Плохо -->
<img src="chart.png" alt="">

<!-- Хорошо -->
<img src="chart.png" alt="Диаграмма показывает рост продаж в 2023 году">
```

### Уровень AA (Средний)
Средний уровень доступности. Игнорирование этих критериев создает барьеры для многих пользователей с ограниченными возможностями.

**Пример:**
```css
/* Обеспечение достаточного контраста текста */
.text-high-contrast {
  color: #000000;
  background-color: #ffffff;
  /* Соотношение контраста 21:1 */
}

.text-medium-contrast {
  color: #333333;
  background-color: #ffffff;
  /* Соотношение контраста 12.6:1 - минимальный стандарт AA */
}
```

### Уровень AAA (Повышенный)
Повышенный уровень доступности. Игнорирование этих критериев создает барьеры для некоторых пользователей с ограниченными возможностями.

## Критерии успеха

Каждый уровень содержит конкретные критерии успеха, которые являются тестопригодными утверждениями, определяющими требования доступности. Например:

- 1.1.1 Нетекстовое содержимое (A)
- 1.2.1 Аудио и видео (A)
- 1.3.1 Информация и отношения (A)
- 1.4.3 Контраст (AA)
- 1.4.4 Изменение размера текста (AA)

## Практические рекомендации

1. **Интеграция в процесс разработки**
   - Включите требования WCAG в пользовательские истории
   - Проводите регулярные проверки доступности на этапе разработки
   - Обучайте команду основам доступности

2. **Тестирование**
   - Используйте автоматизированные инструменты (axe, WAVE)
   - Проводите ручное тестирование
   - Тестируйте с вспомогательными технологиями (скринридеры, клавиатура)

3. **Проверка соответствия**
   - Выполняйте аудиты доступности регулярно
   - Учитывайте юридические требования (например, ADA, EN 301 549)

## Заключение

WCAG предоставляют всестороннюю основу для создания доступного веб-контента. Хотя достижение полного соответствия может потребовать значительных усилий, даже частичное применение этих принципов значительно улучшит опыт пользователей с ограниченными возможностями.

> [!tip] Совет
> Начните с внедрения принципов WCAG на ранних этапах разработки, это значительно сокращает время и ресурсы, необходимые для исправления проблем доступности позже.

> [!warning] Важно
> WCAG - это руководящие принципы, а не закон. Однако во многих юрисдикциях они становятся частью законодательства о доступности.
