# Основы веб-безопасности

## Введение в веб-безопасность

Веб-безопасность — это область информационной безопасности, сосредоточенная на защите веб-приложений, веб-сайтов и веб-сервисов от различных угроз и атак. С ростом зависимости от интернет-технологий и увеличением объема конфиденциальных данных, обрабатываемых веб-приложениями, важность веб-безопасности значительно возросла.

Веб-безопасность включает в себя комплекс мер, направленных на защиту:
- Данных пользователей
- Инфраструктуры серверов
- Целостности приложений
- Конфиденциальности информации

> [!info] Зачем нужна веб-безопасность?
> Современные веб-приложения обрабатывают огромное количество чувствительной информации, включая личные данные, платежные реквизиты и корпоративную информацию. Нарушение безопасности может привести к серьезным финансовым и репутационным потерям.

## Основные понятия и терминология

### Уязвимость (Vulnerability)
Слабое место в системе, которое может быть использовано злоумышленником для нарушения безопасности.

### Угроза (Threat)
Потенциальный источник вреда, который может воспользоваться уязвимостью.

### Атака (Attack)
Конкретное действие или серия действий, направленных на эксплуатацию уязвимости.

### Модель угроз (Threat Model)
Систематический подход к идентификации, пониманию и документированию потенциальных угроз.

### CIA (Confidentiality, Integrity, Availability)
Три основных принципа информационной безопасности:
- **Конфиденциальность** — защита от несанкционированного доступа
- **Целостность** — обеспечение точности и полноты данных
- **Доступность** — обеспечение доступности информации при необходимости

## Модели угроз

### STRIDE
Модель угроз от Microsoft, охватывающая шесть основных типов угроз:
- **Spoofing** (подмена) — подмена личности пользователя
- **Tampering** (модификация) — несанкционированное изменение данных
- **Repudiation** (отрицание) — отрицание выполнения операции
- **Information Disclosure** (раскрытие информации) — доступ к конфиденциальным данным
- **Denial of Service** (отказ в обслуживании) — недоступность сервиса
- **Elevation of Privilege** (повышение привилегий) — получение несанкционированных прав

### DREAD
Модель оценки рисков, используемая вместе с STRIDE:
- **Damage** (ущерб) — насколько серьезен ущерб
- **Reproducibility** (воспроизводимость) — насколько легко воспроизвести атаку
- **Exploitability** (эксплуатируемость) — насколько легко эксплуатировать
- **Affected users** (затронутые пользователи) — сколько пользователей может быть затронуто
- **Discoverability** (обнаруживаемость) — насколько легко обнаружить уязвимость

## Основные принципы безопасности

### Принцип наименьших привилегий
Пользователи и процессы должны иметь минимально необходимые права для выполнения своих функций.

### Защита в глубину (Defense in Depth)
Использование нескольких уровней защиты для обеспечения безопасности даже при сбое одного уровня.

### Безопасность по умолчанию
Система должна быть безопасной по умолчанию, даже если администраторы ничего не настраивают.

### Принцип отказа
Система должна безопасно отказываться в случае ошибок, не раскрывая чувствительную информацию.

### Проверка всех входных данных
Все входные данные должны быть проверены и санитизированы перед обработкой.

## Распространенные уязвимости

### 1. Injection (Внедрение)
Когда необработанные данные отправляются в интерпретатор в качестве команды или запроса.

**Примеры:**
- SQL Injection
- Command Injection
- LDAP Injection
- XPath Injection

**Профилактика:**
- Использование подготовленных выражений
- Санитизация входных данных
- Принцип наименьших привилегий для баз данных

### 2. Broken Authentication (Нарушенная аутентификация)
Уязвимости, связанные с процессом аутентификации и управления сессиями.

**Примеры:**
- Предсказуемые токены сессии
- Отсутствие ограничения попыток входа
- Хранение паролей в открытом виде

**Профилактика:**
- Использование надежных механизмов аутентификации
- Реализация двухфакторной аутентификации
- Корректное управление сессиями

### 3. Sensitive Data Exposure (Раскрытие чувствительных данных)
Недостаточная защита чувствительных данных, таких как финансовая информация, персональные данные и пароли.

**Примеры:**
- Хранение паролей без хеширования
- Передача данных по незащищенным каналам
- Неправильная конфигурация SSL/TLS

**Профилактика:**
- Шифрование чувствительных данных
- Использование HTTPS для всех передач данных
- Применение надежных алгоритмов хеширования

### 4. XML External Entities (XXE)
Уязвимости, возникающие при обработке XML-документов с внешними сущностями.

**Примеры:**
- Чтение локальных файлов
- Сканирование внутренней сети
- Запуск атак типа "отказ в обслуживании"

**Профилактика:**
- Отключение обработки внешних сущностей
- Использование менее сложных форматов данных (JSON)
- Санитизация XML-входов

### 5. Broken Access Control (Нарушенный контроль доступа)
Недостаточная проверка прав доступа к функциям и данным.

**Примеры:**
- Прямые ссылки на защищенные ресурсы
- Отсутствие проверки прав на серверной стороне
- Неправильная конфигурация ролей

**Профилактика:**
- Серверная проверка всех запросов
- Использование ролевой модели доступа
- Регулярный аудит прав доступа

### 6. Security Misconfiguration (Неправильная конфигурация безопасности)
Небезопасные настройки системы, сервера или приложения.

**Примеры:**
- Открытые порты
- Стандартные учетные записи
- Отладочная информация в продакшене

**Профилактика:**
- Регулярное обновление конфигураций
- Удаление ненужных компонентов
- Использование безопасных настроек по умолчанию

### 7. Cross-Site Scripting (XSS)
Уязвимость, позволяющая внедрять вредоносный скрипт в веб-страницу, просматриваемую другими пользователями.

**Типы XSS:**
- **Reflected XSS** — скрипт отражается от сервера
- **Stored XSS** — скрипт сохраняется на сервере
- **DOM-based XSS** — уязвимость на стороне клиента

**Профилактика:**
- Экранирование вывода
- Санитизация входных данных
- Использование Content Security Policy (CSP)

### 8. Insecure Deserialization
Уязвимость, возникающая при десериализации данных из ненадежного источника.

**Примеры:**
- Выполнение произвольного кода
- Атаки типа "отказ в обслуживании"
- Обход аутентификации

**Профилактика:**
- Проверка целостности данных
- Использование безопасных форматов сериализации
- Ограничение типов десериализуемых объектов

### 9. Using Components with Known Vulnerabilities (Использование компонентов с известными уязвимостями)
Использование библиотек и фреймворков с известными уязвимостями.

**Профилактика:**
- Регулярное обновление зависимостей
- Использование инструментов анализа уязвимостей
- Мониторинг CVE и безопасности компонентов

### 10. Insufficient Logging & Monitoring (Недостаточное логирование и мониторинг)
Отсутствие адекватного логирования и мониторинга для обнаружения атак.

**Профилактика:**
- Логирование всех важных событий
- Настройка оповещений о подозрительной активности
- Регулярный аудит логов
