# Защита от браузерных атак

## Введение в атаки на уровне браузера

Атаки на уровне браузера представляют собой класс уязвимостей и методов эксплуатации, которые направлены на клиентскую часть веб-приложений - браузер пользователя. Эти атаки отличаются от серверных атак тем, что злоумышленник манипулирует поведением браузера пользователя, чтобы получить несанкционированный доступ к данным, выполнить нежелательные действия или обойти защитные механизмы.

Браузерные атаки особенно опасны, потому что они могут обойти традиционные серверные меры безопасности, полагаясь на доверие пользователя к веб-сайту и возможности JavaScript в браузере. Современные веб-приложения, которые активно используют JavaScript и сложные клиентские фреймворки, становятся все более уязвимыми к этим типам атак.

> [!warning] Важно
> С развитием одностраничных приложений (SPA) и богатых клиентских интерфейсов, защита от браузерных атак становится критически важной для обеспечения безопасности веб-приложений.

## Типы атак на уровне браузера

Существует множество различных типов атак на уровне браузера, каждая из которых имеет свои особенности эксплуатации и методы защиты:

### 1. Клиентские атаки инъекций
- [[XSS-атаки]] - внедрение вредоносного скрипта в доверенный сайт
- [[DOM-инъекции]] - манипуляции с DOM, приводящие к выполнению вредоносного кода
- [[Client-side template injection]] - внедрение шаблонов на стороне клиента

### 2. Атаки на движок JavaScript
- [[Prototype pollution]] - манипуляции с прототипами объектов
- [[Type confusion attacks]] - атаки на основе неправильной типизации
- [[Memory corruption in JavaScript engines]] - уязвимости в движках JavaScript

### 3. Атаки на браузерные API
- [[Browser-side rendering attacks]] - атаки на механизмы отображения
- [[Cross-site request forgery]] - подделка межсайтовых запросов
- [[Clickjacking]] - захват кликов пользователя

### 4. Атаки на расширения браузера
- [[Content security bypass in extensions]] - обход политик безопасности
- [[Extension privilege escalation]] - повышение привилегий в расширениях

Каждый тип атаки требует специфического подхода к защите, и понимание различий между ними критически важно для разработки эффективных мер защиты.

## DOM-инъекции

DOM-инъекции (DOM-based injection) представляют собой подкатегорию XSS-атак, при которой вредоносный скрипт внедряется и выполняется исключительно в браузере пользователя без участия веб-сервера. В отличие от классических XSS-атак, DOM-инъекции происходят полностью на стороне клиента, что делает их особенно сложными для обнаружения и предотвращения.

### Механизмы DOM-инъекций

DOM-инъекции происходят, когда веб-приложение использует непроверенные данные из URL или других источников для динамического изменения DOM. Основные механизмы:

1. **Использование window.location**: Приложение использует части URL (хэш, параметры) для изменения содержимого страницы
2. **Небезопасная работа с DOM-методами**: Использование методов, таких как `innerHTML`, `outerHTML`, `document.write` с непроверенными данными
3. **Манипуляции с атрибутами**: Установка значений атрибутов (например, `href`, `src`) на основе пользовательского ввода
4. **Шаблонизация на клиенте**: Использование шаблонизаторов с непроверенными данными

### Примеры DOM-инъекций

```javascript
// Небезопасный код
function updateContent() {
    var hash = window.location.hash.substr(1);
    document.getElementById("content").innerHTML = hash;
}

// При переходе по ссылке:
// http://example.com/page.html#<script>alert('XSS')</script>
// Скрипт выполнится в браузере пользователя
```

```javascript
// Еще один пример
function redirectUser() {
    var redirect = getUrlParameter('redirect');
    window.location.href = redirect;
}

// Возможность перенаправления на вредоносный сайт
```

### Методы обнаружения DOM-инъекций

1. **Анализ JavaScript-кода**: Ручной или автоматический поиск потенциально опасных функций
2. **Использование инструментов статического анализа**: ESLint с правилами безопасности, JSHint
3. **Динамическое тестирование**: Проверка различных входных данных в браузере
4. **Использование специализированных инструментов**: DOM XSS Scanner, CSP нарушения

### Меры защиты от DOM-инъекций

1. **Избегайте прямой вставки пользовательского ввода в DOM**: Используйте `textContent` вместо `innerHTML`
2. **Санитизация данных**: Проверяйте и очищайте все данные перед использованием
3. **Использование безопасных API**: Предпочтительно использовать безопасные методы манипуляции DOM
4. **Content Security Policy**: Ограничение выполнения встроенных скриптов
5. **Кодирование специальных символов**: Преобразование специальных символов в безопасные эквиваленты

> [!tip] Лучшая практика
> Используйте фреймворки, которые автоматически экранируют данные (например, React, Vue), так как они значительно снижают риск DOM-инъекций.

## Prototype pollution

Prototype pollution - это тип уязвимости, характерный для языка JavaScript, при котором злоумышленник может манипулировать прототипами объектов, изменяя поведение приложения. Эта уязвимость особенно актуальна в современных веб-приложениях, где активно используется динамическая манипуляция объектами.

### Суть атаки prototype pollution

В JavaScript все объекты наследуют свойства и методы от прототипов. Атака prototype pollution позволяет злоумышленнику добавлять, изменять или удалять свойства в прототипах, что может повлиять на поведение всего приложения. Это особенно опасно, когда приложение использует рекурсивное объединение объектов или десериализацию данных из внешних источников.

### Механизмы эксплуатации

```javascript
// Уязвимый код
function merge(target, source) {
    for (let key in source) {
        if (key in target) {
            merge(target[key], source[key]);
        } else {
            target[key] = source[key];
        }
    }
}

// Злоумышленник может передать:
const payload = JSON.parse('{"constructor": {"prototype": {"polluted": true}}}');
merge({}, payload);

// Теперь все новые объекты будут иметь свойство polluted: true
```

### Последствия prototype pollution

1. **Обход безопасности**: Изменение свойств, отвечающих за проверку авторизации
2. **Изменение поведения приложения**: Манипуляции с логикой работы приложения
3. **Выполнение произвольного кода**: В некоторых случаях может привести к RCE
4. **Нарушение целостности данных**: Изменение критических свойств объектов

### Примеры уязвимого кода

```javascript
// Небезопасное объединение объектов
function extend(target, source) {
    for (let prop in source) {
        target[prop] = source[prop];
    }
    return target;
}

// Уязвимый вызов
extend(userConfig, userInput);
```

```javascript
// Использование рекурсивного объединения
function deepExtend(target, source) {
    for (let key in source) {
        if (typeof source[key] === 'object' && source[key] !== null) {
            target[key] = deepExtend(target[key] || {}, source[key]);
        } else {
            target[key] = source[key];
        }
    }
    return target;
}
```

### Методы обнаружения

1. **Анализ исходного кода**: Поиск потенциально уязвимых мест в коде
2. **Использование статических анализаторов**: ESLint с плагинами безопасности, NodeJsScan
3. **Динамическое тестирование**: Проверка с использованием полезных нагрузок prototype pollution
4. **Мониторинг изменений прототипов**: Использование Object.preventExtensions() и других методов

### Меры защиты

1. **Проверка ключей объекта**: Избегайте использования ключей "constructor" и "__proto__"
2. **Использование безопасных методов объединения**: Используйте проверенные библиотеки (например, lodash с фиксами)
3. **Ограничение доступа к прототипам**: Использование Object.freeze() для критических прототипов
4. **Валидация входных данных**: Проверка структуры и содержимого объектов перед объединением
5. **Content Security Policy**: Ограничение возможностей выполнения JavaScript

> [!danger] Опасность
> Prototype pollution может привести к серьезным последствиям, включая обход аутентификации и выполнение произвольного кода. Особенно опасно в серверных приложениях на Node.js.

## Client-side template injection

Client-side template injection (CSTI) - это уязвимость, возникающая при внедрении вредоносного шаблона в клиентский шаблонизатор. В отличие от серверного шаблонного инъекции, CSTI происходит полностью на стороне клиента и может обойти серверные меры защиты.

### Основные характеристики CSTI

Client-side template injection возникает, когда приложение динамически рендерит шаблоны, используя пользовательский ввод. Это может происходить в различных фреймворках и библиотеках, таких как Angular, Vue, Handlebars, и других.

### Механизмы эксплуатации

```html
<!-- Уязвимое использование шаблонизатора -->
<div id="content">{{ userInput }}</div>

<!-- Если userInput содержит: {{7*7}} -->
<!-- Результат: 49 (шаблон выполнится) -->
```

```javascript
// Использование шаблонизатора на клиенте
var template = Handlebars.compile(userTemplate);
var result = template(userContext);
document.getElementById('output').innerHTML = result;
```

### Примеры уязвимостей в популярных фреймворках

#### AngularJS (устаревший, но часто встречается)

```html
<!-- Уязвимый код -->
<div ng-app ng-csp>
    <p>{{ userInput }}</p>
</div>
```

```javascript
// Уязвимый вызов
$scope.userInput = "{{constructor.constructor('alert(1)')()}}";
```

#### Vue.js

```html
<!-- Уязвимый код -->
<div id="app">{{ userInput }}</div>
```

```javascript
// При неправильной конфигурации может быть уязвимым
new Vue({
    el: '#app',
    data: {
        userInput: userInputFromServer
    }
});
```

### Последствия CSTI

1. **Выполнение произвольного JavaScript**: Злоумышленник может выполнить вредоносный код в контексте браузера
2. **Кража данных пользователя**: Доступ к cookies, сессионным данным, личной информации
3. **Манипуляции с DOM**: Изменение содержимого страницы, перенаправление пользователя
4. **Обход CSP**: В некоторых случаях может обойти Content Security Policy

### Методы обнаружения

1. **Тестирование специфических полезных нагрузок**: Попытка ввести синтаксис шаблонизатора
2. **Анализ исходного кода**: Поиск мест, где пользовательский ввод используется в шаблонах
3. **Использование автоматизированных инструментов**: CSP нарушения, специализированные сканеры
4. **Динамическое тестирование**: Проверка различных синтаксисов шаблонизаторов

### Меры защиты

1. **Избегайте динамической компиляции шаблонов**: Не используйте пользовательский ввод в шаблонах
2. **Санитизация шаблонов**: Проверяйте и очищайте пользовательский ввод
3. **Использование белых списков**: Разрешайте только безопасные теги и атрибуты
4. **Content Security Policy**: Ограничение выполнения встроенных скриптов
5. **Конфигурация шаблонизаторов**: Использование безопасных настроек шаблонизаторов

> [!tip] Лучшая практика
> Используйте шаблонизаторы, которые автоматически экранируют выражения, и избегайте динамической компиляции шаблонов с пользовательским вводом.

## Browser-side rendering (BSR) attacks

Browser-side rendering (BSR) attacks - это класс атак, направленных на механизмы рендеринга браузера, особенно в контексте современных фреймворков, использующих виртуальный DOM и динамическую загрузку контента. Эти атаки эксплуатируют особенности клиентского рендеринга для выполнения вредоносных действий.

### Суть BSR-атак

BSR-атаки отличаются от традиционных XSS-атак тем, что они эксплуатируют специфические особенности работы современных фреймворков и библиотек рендеринга. Эти атаки могут обойти традиционные защиты, поскольку происходят в контексте обновления DOM через фреймворк.

### Типы BSR-атак

1. **Атаки на виртуальный DOM**: Манипуляции с виртуальным DOM для обхода проверок
2. **Атаки на механизм обновления**: Эксплуатация особенностей diff-алгоритмов
3. **Атаки на жизненные циклы компонентов**: Использование хуков жизненного цикла для выполнения кода
4. **Атаки на динамическую загрузку**: Эксплуатация lazy loading и динамических импортов

### Примеры BSR-атак

```javascript
// Пример атаки через механизм обновления в React
function VulnerableComponent({ userInput }) {
    // Предположим, что userInput проходит через какой-то фильтр
    return <div dangerouslySetInnerHTML={{__html: userInput}} />;
}

// Даже с фильтрацией, BSR-атаки могут обойти защиту
// через особенности рендеринга или обновления компонентов
```

```javascript
// Пример атаки через жизненные циклы
class VulnerableComponent extends React.Component {
    componentDidMount() {
        // Если здесь используется непроверенный пользовательский ввод
        const script = this.props.userScript;
        eval(script); // Очень опасно!
    }
}
```

### Механизмы эксплуатации

1. **Обход санитизации**: Использование особенностей обновления DOM
2. **Эксплуатация жизненных циклов**: Выполнение кода в хуках жизненного цикла
3. **Манипуляции с состоянием**: Изменение состояния компонентов для обхода защиты
4. **Обход CSP**: Использование особенностей фреймворков для обхода политики безопасности

### Последствия BSR-атак

1. **Обход традиционных XSS-защит**: Фреймворк может не распознать атаку
2. **Выполнение произвольного кода**: В обход Content Security Policy
3. **Нарушение целостности приложения**: Изменение логики работы компонентов
4. **Кража данных**: Доступ к данным, хранящимся в состоянии приложения

### Методы обнаружения

1. **Анализ кода фреймворков**: Изучение особенностей конкретных фреймворков
2. **Тестирование обновлений DOM**: Проверка поведения при динамическом обновлении
3. **Мониторинг CSP нарушений**: Отслеживание попыток обхода политики безопасности
4. **Статический анализ компонентов**: Проверка на наличие потенциально уязвимого кода

### Меры защиты от BSR-атак

1. **Ограничение использования опасных методов**: Избегайте `dangerouslySetInnerHTML`, `eval`, и т.д.
2. **Валидация данных перед рендерингом**: Проверка всех данных, используемых в компонентах
3. **Использование обновлений только через безопасные API**: Работа только с API фреймворка
4. **Комплексная политика CSP**: Ограничение всех потенциально опасных возможностей
5. **Тестирование на BSR-уязвимости**: Регулярное тестирование с использованием специфических полезных нагрузок

> [!warning] Важно
> BSR-атаки особенно опасны в современных SPA-приложениях, где большая часть логики выполняется на клиенте.

## Уязвимости в браузерных расширениях

Браузерные расширения предоставляют мощные API для взаимодействия с веб-страницами, что делает их потенциальным источником уязвимостей. Эти уязвимости могут быть использованы как для атак на пользователя, так и для атак на веб-сайты, которые пользователь посещает.

### Типы уязвимостей в расширениях

1. **Небезопасная обработка контента**: Расширения могут обрабатывать вредоносный контент на веб-страницах
2. **Неправильные разрешения**: Расширения могут запрашивать больше прав, чем необходимо
3. **Уязвимости в фоновых скриптах**: Скрипты, выполняющиеся в фоне, могут содержать уязвимости
4. **Небезопасная коммуникация**: Передача данных между компонентами расширения может быть небезопасной
5. **Уязвимости в манифесте**: Неправильная конфигурация манифеста может привести к уязвимостям

### Примеры уязвимостей

```json
// Небезопасный манифест расширения
{
    "manifest_version": 3,
    "name": "My Extension",
    "permissions": [
        "activeTab",
        "*://*/*"  // Слишком широкое разрешение
    ],
    "content_scripts": [{
        "matches": ["<all_urls>"], // Все URL
        "js": ["content.js"]
    }]
}
```

```javascript
// Небезопасный контент-скрипт
function processContent(content) {
    // Небезопасная вставка содержимого
    document.body.innerHTML += content;
}
```

### Возможные атаки через расширения

1. **XSS через расширения**: Расширение может вставлять вредоносный скрипт на веб-страницу
2. **Кража данных**: Расширение может собирать данные с посещаемых пользователем сайтов
3. **Манипуляции с запросами**: Расширение может изменять HTTP-запросы и ответы
4. **Обход CORS**: Расширения могут обходить политику CORS
5. **Привилегированный доступ**: Расширения имеют доступ к API, недоступным для веб-страниц

### Меры защиты при разработке расширений

1. **Минимизация разрешений**: Запрашивайте только необходимые разрешения
2. **Санитизация ввода**: Проверяйте и очищайте все данные, обрабатываемые расширением
3. **Избегайте eval и подобных функций**: Не выполняйте произвольный код
4. **Безопасная коммуникация**: Используйте безопасные методы передачи данных между компонентами
5. **Проверка контента**: Проверяйте содержимое веб-страниц перед обработкой

### Безопасность для пользователей

1. **Проверка разработчиков**: Устанавливайте расширения только от проверенных разработчиков
2. **Проверка разрешений**: Внимательно изучайте запрашиваемые разрешения
3. **Регулярные обновления**: Поддерживайте расширения в актуальном состоянии
4. **Мониторинг поведения**: Следите за необычной активностью расширений

> [!danger] Опасность
> Злонамеренные расширения могут получить доступ ко всем веб-сайтам, которые посещает пользователь, что делает их особенно опасными.

## Межсайтовые атаки на браузер

Межсайтовые атаки на браузер включают в себя различные типы атак, которые эксплуатируют доверие браузера к различным сайтам и доменам. Эти атаки могут происходить как между сайтами, так и внутри одного сайта с разными поддоменами.

### Типы межсайтовых атак

1. **Cross-Site Scripting (XSS)**: Внедрение вредоносного скрипта с одного сайта на другой
2. **Cross-Site Request Forgery (CSRF)**: Принуждение пользователя к выполнению нежелательных действий
3. **Cross-Site Script Inclusion (XSSI)**: Внедрение скрипта для получения доступа к данным
4. **Cross-Origin Resource Sharing (CORS) bypass**: Обход политик CORS
5. **PostMessage vulnerabilities**: Уязвимости в междоменной коммуникации

### Cross-Site Script Inclusion (XSSI)

XSSI - это атака, при которой злоумышленник включает скрипт с другого домена, чтобы получить доступ к данным, которые не должны быть доступны.

```javascript
// Уязвимый JavaScript-файл на сайте victim.com
var user_data = {
    "name": "John Doe",
    "email": "john@example.com",
    "balance": 1000
};

// Вредоносная страница на evil.com
<script>
var originalJSONParse = JSON.parse;
JSON.parse = function() {
    alert('XSSI: ' + arguments[0]);
    return originalJSONParse.apply(this, arguments);
};

// Включение скрипта с victim.com
var script = document.createElement('script');
script.src = 'https://victim.com/user_data.js';
document.head.appendChild(script);
</script>
```

### PostMessage уязвимости

PostMessage позволяет безопасно обмениваться данными между окнами/фреймами разных доменов, но неправильная реализация может привести к уязвимостям:

```javascript
// Небезопасный прием сообщений
window.addEventListener('message', function(event) {
    // НЕ проверяем источник!
    document.getElementById('content').innerHTML = event.data;
});

// Безопасная реализация
window.addEventListener('message', function(event) {
    if (event.origin !== 'https://trusted.com') {
        return;
    }
    document.getElementById('content').innerHTML = event.data;
});
```

### Меры защиты от межсайтовых атак

1. **Content Security Policy**: Ограничение источников выполнения скриптов
2. **SameSite атрибуты куки**: Ограничение отправки куки между сайтами
3. **Проверка источников в PostMessage**: Всегда проверяйте origin
4. **CORS политики**: Правильная настройка политик CORS
5. **CSRF-токены**: Защита от межсайтовой подделки запросов

> [!tip] Лучшая практика
> Используйте комплексный подход к защите, включая несколько уровней защиты от межсайтовых атак.

## Защита от атак на движок браузера

Современные браузеры содержат сложные движки рендеринга и JavaScript, которые могут содержать уязвимости. Защита от атак на движок браузера включает как меры на стороне разработчика, так и меры на стороне пользователя.

### Типы уязвимостей в движках браузера

1. **Уязвимости в JavaScript-движках**: JIT-компиляция, оптимизации, GC
2. **Уязвимости в рендеринге**: Обработка HTML, CSS, изображений
3. **Уязвимости в парсерах**: HTML, CSS, XML парсеры
4. **Уязвимости в Web APIs**: Неправильная реализация браузерных API
5. **Уязвимости в WebAssembly**: Потенциальные проблемы с безопасностью WebAssembly

### Примеры уязвимостей в движках

1. **Use-after-free (UAF)**: Использование памяти после освобождения
2. **Type confusion**: Неправильная типизация объектов
3. **Integer overflow/underflow**: Проблемы с арифметикой
4. **Race conditions**: Условия гонки в многопоточной среде
5. **JIT spray**: Атаки на JIT-компиляторы

### Меры защиты от атак на движок

1. **Изоляция процессов**: Современные браузеры изолируют рендеринг в отдельных процессах
2. **Sandbox**: Ограничение возможностей рендер-процессов
3. **ASLR и DEP**: Защита от эксплуатации уязвимостей в памяти
4. **CSP**: Ограничение возможностей вредоносных скриптов
5. **Обновления безопасности**: Регулярные обновления браузеров

### Защита разработчиков приложений

1. **Ограничение возможностей JavaScript**: Использование CSP для ограничения выполнения
2. **Избегание небезопасных API**: Не использовать уязвимые или экспериментальные API
3. **Проверка пользовательского контента**: Санитизация всего пользовательского контента
4. **Тестирование на совместимость**: Проверка работы в разных браузерах

> [!warning] Важно
> Защита от атак на движок браузера требует постоянного обновления знаний о новых уязвимостях и методах защиты.

## Лучшие практики

Для обеспечения максимальной защиты от браузерных атак необходимо следовать комплексному подходу, включающему несколько уровней защиты.

### Общие принципы безопасности

1. **Принцип наименьших привилегий**: Предоставляйте только необходимые права и доступы
2. **Защита по глубине**: Используйте несколько уровней защиты
3. **Проверка и санитизация всего ввода**: Всегда проверяйте и очищайте данные
4. **Контроль доступа**: Ограничивайте доступ к чувствительным функциям
5. **Ведение логов и мониторинг**: Отслеживайте подозрительную активность

### Конкретные рекомендации

#### Для разработчиков фронтенда:

1. **Используйте проверенные фреймворки**: Используйте фреймворки с встроенной защитой
2. **Избегайте unsafe методов**: Не используйте `eval`, `innerHTML`, `document.write` с пользовательским вводом
3. **Валидация данных**: Проверяйте все данные на клиенте и сервере
4. **Корректная обработка событий**: Правильно обрабатывайте пользовательские события
5. **Обновление зависимостей**: Регулярно обновляйте библиотеки и фреймворки

#### Для разработчиков бэкенда:

1. **Content Security Policy**: Реализуйте строгую CSP
2. **Заголовки безопасности**: Используйте все доступные HTTP-заголовки безопасности
3. **Санитизация вывода**: Экранируйте все данные перед отправкой клиенту
4. **Контроль сессий**: Правильно управляйте сессиями и аутентификацией
5. **Валидация API**: Проверяйте все входные данные в API

#### Для пользователей:

1. **Обновление браузера**: Поддерживайте браузер в актуальном состоянии
2. **Осторожность с расширениями**: Устанавливайте только проверенные расширения
3. **Безопасные привычки**: Не переходите по подозрительным ссылкам
4. **Использование антивирусов**: Используйте антивирусное ПО с веб-фильтрацией
5. **Регулярная проверка разрешений**: Проверяйте, какие сайты имеют доступ к камере, микрофону и другим ресурсам

### Автоматизированные инструменты

1. **ESLint с плагинами безопасности**: Проверка JavaScript-кода на уязвимости
2. **Content Security Policy validator**: Проверка корректности CSP
3. **SAST инструменты**: Статический анализ исходного кода
4. **DAST инструменты**: Динамическое тестирование веб-приложений
5. **Dependency checkers**: Проверка уязвимостей в зависимостях

> [!tip] Профессиональный совет
> Регулярно проводите пентестинг и код-ревью для выявления потенциальных уязвимостей, которые могут быть пропущены автоматическими инструментами.
