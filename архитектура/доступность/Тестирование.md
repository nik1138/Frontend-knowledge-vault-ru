# Тестирование доступности веб-приложений

## Обзор

Тестирование доступности - это процесс проверки веб-приложений на соответствие требованиям доступности для пользователей с различными ограничениями здоровья. В российском контексте 2025 года тестирование доступности стало обязательным требованием для государственных и многих коммерческих сайтов, подпадающих под действие закона №419-ФЗ и ГОСТ Р 52872-2012.

## Виды тестирования доступности

### 1. Автоматизированное тестирование
Проверка с использованием инструментов и библиотек, которые могут автоматически выявить большинство распространенных проблем доступности.

#### Инструменты автоматизированного тестирования:

**axe-core**
- Интегрируется в процесс разработки
- Поддержка всех основных браузеров
- Высокая точность обнаружения проблем
- Поддержка WCAG 2.1 AA

**Lighthouse**
- Встроен в Chrome DevTools
- Проверка доступности как часть общего аудита
- Отчет с рекомендациями по улучшению

**Pa11y**
- Командная строка и Node.js API
- Поддержка различных стандартов доступности
- Возможность настройки под конкретные требования

**WAVE**
- Веб-инструмент и браузерное расширение
- Визуальное отображение проблем доступности
- Поддержка русского интерфейса

### 2. Ручное тестирование
Проверка, требующая участия человека, особенно для оценки контекста, качества альтернативного текста и пользовательского опыта.

#### Аспекты ручного тестирования:
- Проверка навигации с клавиатуры
- Тестирование с вспомогательными технологиями
- Оценка качества альтернативного текста
- Проверка контрастности цветов
- Оценка структуры и логики контента

### 3. Тестирование с вспомогательными технологиями
Использование специализированных программ и устройств, которые помогают пользователям с ограниченными возможностями взаимодействовать с веб-контентом.

## Инструменты и технологии

### Скринридеры (Технологии чтения с экрана)

**NVDA (NonVisual Desktop Access)**
- Бесплатный скринридер, популярный в России
- Поддержка русского языка
- Совместимость с Firefox и другими браузерами
- Горячие клавиши: 
  - NVDA + пробел - переключение NVDA
  - NVDA + F7 - список элементов страницы
  - NVDA + Tab - чтение элемента под фокусом

**JAWS (Job Access With Speech)**
- Коммерческий скринридер
- Наиболее распространенный в корпоративной среде
- Поддержка русского языка
- Совместимость с Internet Explorer, Chrome, Firefox

**VoiceOver**
- Встроенный скринридер в macOS и iOS
- Работает на устройствах Apple
- Поддержка русского языка
- Горячие клавиши: Cmd + F5 (на Mac)

### Увеличение и другие вспомогательные технологии

**ZoomText**
- Программа для увеличения и изменения цвета экрана
- Полезна для пользователей с нарушениями зрения

**Dragon NaturallySpeaking**
- Программа распознавания речи
- Используется для управления компьютером голосом

## Методики тестирования

### 1. Тестирование с отключенной мышью
- Отключите мышь или сенсорное управление
- Навигируйте по сайту только с клавиатуры
- Проверьте:
  - Все интерактивные элементы доступны
  - Порядок фокуса логичен
  - Состояние фокуса видимо
  - Все функции доступны без мыши

### 2. Тестирование с увеличением экрана
- Увеличьте масштаб до 200%
- Проверьте, что контент остается читаемым
- Убедитесь, что не появляется горизонтальная прокрутка
- Проверьте, что элементы не перекрывают друг друга

### 3. Тестирование с цветовыми ограничениями
- Используйте инструменты симуляции цветовой слепоты
- Проверьте, что информация не передается только через цвет
- Убедитесь в достаточном контрасте текста и фона

### 4. Тестирование с вспомогательными технологиями
- Проверьте основные сценарии использования с NVDA или JAWS
- Убедитесь, что все интерактивные элементы правильно озвучиваются
- Проверьте структуру заголовков и навигацию по ним

## Критерии успеха (WCAG 2.1 AA)

### Уровень A
- Все не-тестовые элементы мультимедиа имеют альтернативы
- Контент можно представить в разных формах
- Контент можно увеличить до 200%
- Все функции доступны с клавиатуры
- Не вызываются судороги при использовании

### Уровень AA
- Контрастность текста 4.5:1 (3:1 для крупного текста)
- Метки для всех полей ввода
- Заголовки для таблиц
- Понятные цели ссылок
- Понятные заголовки разделов

## Практические рекомендации

### 1. Интеграция в процесс разработки
```javascript
// Пример интеграции axe-core в тесты
const axe = require('axe-core');

// Проверка доступности в Jest тестах
test('страница соответствует требованиям доступности', async () => {
  const results = await axe.run(document.body);
  expect(results.violations).toHaveLength(0);
});
```

```bash
# Пример использования Pa11y в CI/CD
npx pa11y https://example.com --standard WCAG2AA
```

### 2. Проверка контрастности цветов
- Используйте инструменты для проверки контрастности:
  - WebAIM Contrast Checker
  - Chrome DevTools
  - Color Oracle (симулятор цветовой слепоты)

### 3. Проверка альтернативного текста
- Альтернативный текст должен быть информативным и кратким
- Для декоративных изображений используйте пустой alt=""
- Для сложных изображений (диаграммы, графики) предоставьте подробное описание

### 4. Тестирование форм
- Проверьте, что все поля имеют метки
- Убедитесь, что ошибки валидации правильно сообщаются
- Проверьте, что фокус устанавливается на соответствующие поля при ошибках

## Российские особенности тестирования

В 2025 году в России:
- Увеличилось количество специалистов по доступности
- Повысилась осведомленность о требованиях к доступности
- Российские компании начали внедрять регулярное тестирование доступности
- Появились специализированные компании по тестированию доступности
- Ужесточились требования к доступности государственных сайтов

### Российские инструменты и ресурсы:
- Тестирование с использованием NVDA (наиболее популярный скринридер в России)
- Учет особенностей русского языка при тестировании
- Соответствие российским стандартам (ГОСТ Р 52872-2012)

## Проверка соответствия требованиям

### Федеральный закон №419-ФЗ
- Проверка доступности государственных и муниципальных сайтов
- Обязательная оценка соответствия для организаций с численностью более 100 человек
- Регулярный аудит доступности

### ГОСТ Р 52872-2012
- Проверка эргономических требований
- Оценка доступности для пользователей с различными ограничениями
- Тестирование с использованием вспомогательных технологий

## Отчетность и документирование

### Структура отчета о доступности:
1. Общая информация о тестируемом ресурсе
2. Используемые методы и инструменты
3. Выявленные проблемы
4. Рекомендации по устранению
5. Приоритетность проблем
6. План устранения

### Пример шаблона отчета:
```
Отчет о тестировании доступности
================================

1. Объект тестирования: example.com
2. Дата тестирования: 15.01.2025
3. Методы: автоматизированное тестирование, ручное тестирование, тестирование с NVDA
4. Найденные проблемы:
   - Недостаточная контрастность заголовков (4.2:1, требуется 4.5:1)
   - Отсутствие альтернативного текста у изображений
   - Проблемы с порядком фокуса в форме
5. Рекомендации:
   - Увеличить контрастность текста
   - Добавить атрибуты alt к изображениям
   - Исправить порядок табуляции
6. Приоритеты: высокий, средний, низкий
```

## Заключение

Тестирование доступности - это непрерывный процесс, который должен интегрироваться в весь цикл разработки веб-приложений. В условиях российского законодательства 2025 года, регулярное тестирование доступности становится не только этической необходимостью, но и юридическим требованием. Эффективное тестирование доступности требует сочетания автоматизированных инструментов, ручного тестирования и тестирования с вспомогательными технологиями.
